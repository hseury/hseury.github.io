思考:Android中为什么主线程不会因为Looper.loop()里的死循环卡死

在知乎上看到一个问题：

Android中为什么主线程不会因为Looper.loop()里的死循环卡死？

查看了答案之后，自己总结了下。

在ActivityThread类中，

其中H是ActivityThread对应的Handler,它的handleMessage()方法是这样的

Activity的生命周期都有对应的case条件了，ActivityThread有个getHandler() 方法，得到这个handler就可以发送消息，然后loop 里就分发消息，然后就发给handler，然后就执行到 H(Handler)里的对应代码。所以这些代码就不会卡死，有消息过来就能执行。所有的操作都是执行在Looper中的。Looper.loop()方法中，如果next消息为null，就会return，此时确实会进入block状态，等到有消息时，会被再次唤醒，继续依次执行消息。

Android 的应用是由事件驱动的，looper.loop() 不断地接收事件、处理事件，每一个应用都是运行在 Looper.loop() 的控制之下，如果它停止了，应用也就停止了。只能是某一个应用阻塞 Looper.loop()，而不是 Looper.loop() 阻塞应用。

这是一种常见的事件驱动模型，windows的消息循环机制也类似，可以深入了解这些知识epoll+pipe。

位运算

在看View的绘制的源码时，看到这样的代码。

base/core/java/android/view/View.java

通过位操作来标记boolean变量。

这是因为java中，int类型长度是32位，化成二进制，每一位上是0、1与否，都能代表一个boolean，所以一个int类型可以存储32个boolean值。

Android自定义控件中有个MeasureSpec类，可以将测量模式和size用同一个int表示。

布局

include

该标签的目的是解决重复定义布局的问题而诞生的，提高代码的复用。

merge

使用merge标签可以减少多余的层次结构，多用于一个布局需要包含另一个布局的时候，或者这一个布局用到的重复样式比较多。在加载merge标签的时候，不会加载父控件，只会加载merge标签中的子view。merge标签结合include一起使用可以灵活的、可复用性强的创建布局。

viewstub

ViewStub有一个比较大的优点就是当你用到这个布局的时候才会去加载，例如我们在布局里写了ViewStub并引用一个我们写好的布局。这时候我们可以在我们的activity中使用的时候，必须先设置visible。 这样这个布局才会被加载显示。否则是不会显示的。为了防止都findViewById 方法，我们可以调用inflate()方法，这个方法会返回一个view，我们可以直接对这个view进行操作。
